# Form Rules

Forms MUST use the Composed component hierarchy. Never build form layout with raw HTML elements or inline styles.

## FORM-01 — Required Form Structure

All forms MUST follow this structure:

```
PageLayout
  └─ PageHeader (with backHref for edit/create pages)
  └─ <form onSubmit={handleSubmit}>
       └─ FormFieldSet (legend="Group Name")  ← one per logical group
            └─ FormRow (cols={2})             ← for side-by-side fields
                 └─ FormField (label, required?, description?)
                      └─ Input | Select | Textarea | Checkbox | DateRangePicker
            └─ FormField                      ← single-column field
                 └─ Textarea
       └─ FormFieldSet (legend="Next Group")
            └─ ...
       └─ FormActions
            └─ ActionButton (variant="outline") Cancel
            └─ ActionButton (type="submit") Save
```

**Canonical example:**
```tsx
<PageLayout>
  <PageHeader title="Create Campaign" backHref="/campaigns" />
  <form onSubmit={handleSubmit}>
    <FormFieldSet legend="Basic Info">
      <FormRow cols={2}>
        <FormField label="Name" required>
          <Input name="name" placeholder="Enter name" />
        </FormField>
        <FormField label="Status">
          <Select name="status" options={statusOptions} />
        </FormField>
      </FormRow>
      <FormField label="Description" description="Optional">
        <Textarea name="description" placeholder="Describe the campaign" />
      </FormField>
    </FormFieldSet>
    <FormActions>
      <ActionButton variant="outline" onClick={onCancel}>Cancel</ActionButton>
      <ActionButton type="submit">Save</ActionButton>
    </FormActions>
  </form>
</PageLayout>
```
// WHY: FormFieldSet groups provide visual sectioning without raw div layout. FormRow manages grid spacing. FormField encapsulates label + validation state. This structure is enforced by component types — AI cannot accidentally break it.

**Card-wrapped variant** (when a form section sits within a larger page, not as a standalone form page):
```tsx
<Card>
  <CardHeader><CardTitle>Settings</CardTitle></CardHeader>
  <CardContent>
    <FormFieldSet legend="Preferences">
      <FormField label="Language"><Select options={langs} /></FormField>
    </FormFieldSet>
    <FormActions>
      <ActionButton type="submit">Save</ActionButton>
    </FormActions>
  </CardContent>
</Card>
```

## FORM-02 — FormActions Rules

FormActions MUST always appear:
- At the bottom of the `<form>` element (never inside a FormFieldSet)
- With exactly two ActionButtons: Cancel (outline) + Save (default/submit)
// WHY: Consistent button placement and variant assignment creates muscle memory for users. Cancel is always dismissive (outline), Save is always primary.

```tsx
// CORRECT
<FormActions>
  <ActionButton variant="outline" onClick={handleCancel}>Cancel</ActionButton>
  <ActionButton type="submit" disabled={isSubmitting}>
    {isSubmitting ? "Saving..." : "Save"}
  </ActionButton>
</FormActions>

// FORBIDDEN — reversed order
<FormActions>
  <ActionButton type="submit">Save</ActionButton>
  <ActionButton variant="outline">Cancel</ActionButton>
</FormActions>

// FORBIDDEN — inside FormFieldSet
<FormFieldSet legend="Actions">
  <ActionButton type="submit">Save</ActionButton>
</FormFieldSet>
```

## FORM-03 — Forbidden Form Patterns

```tsx
// FORBIDDEN — div layout for form structure
<form>
  <div style={{ marginBottom: 16 }}>
    <label>Name</label>
    <input />
  </div>
</form>

// FORBIDDEN — bare Input outside FormField
<Card>
  <Input placeholder="Campaign name" />
</Card>

// FORBIDDEN — inline style spacing on form components
<FormField style={{ marginTop: "24px" }} label="Name">
  <Input />
</FormField>
```

For the complete list of forbidden patterns, see: @.claude/rules/forbidden.md

## Validation Patterns

**Required fields:** Use `required` prop on FormField — do not add asterisks manually.
```tsx
<FormField label="Campaign Name" required>  // renders label + * indicator
  <Input name="name" />
</FormField>
```

**Error display:** FormField surfaces validation state via its `error` prop (if the Composed FormField supports it). If the project uses react-hook-form, pass error message to `description` as a fallback.
// WHY: Error messages must appear adjacent to the field, not in a separate alert. FormField's layout guarantees this placement.

**Helper text:** Use `description` prop — never add a separate `<p>` below the input.
```tsx
<FormField label="Budget" description="Maximum spend per day">
  <Input type="number" />
</FormField>
```

For full FormField/FormRow/FormActions interface contracts, see: @.claude/rules/component-interfaces.md

## Escape Hatch

If a form element is genuinely not coverable by FormField (e.g., a custom file uploader, a rich text editor):
1. STOP — do not reach for raw `<input>` or a `<div>` layout
2. Describe the needed element and ask for approval to create a new Composed wrapper
3. After approval, create it in `@/components/composed/` with typed props and no className passthrough
4. Wrap it inside a `FormField` so label, required indicator, and description still apply
