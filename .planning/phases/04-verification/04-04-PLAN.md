  <action>
Execute the automated checker and then fill the mechanical rows of evaluate.md:

**Step 1: Run check-rules.sh**
```bash
bash scripts/check-rules.sh tests/samples/
echo "Exit code: $?"
```

Save the output. For every FAIL line, note which sample file and which check failed.

**Step 2: Open each sample file and fill in the evaluate.md checklist**

For each sample page, read the actual TSX file and fill in the "Actual" and "Pass/Fail" columns for every row in evaluate.md. The rules to check are in the checklist — do not invent new ones.

For rows where check-rules.sh already gave a definitive answer (e.g., FORB-01 inline style check): copy the PASS/FAIL result from the script output.

For rows that require reading the file (e.g., PAGE-04 "ChartSection has cols={2}"): read the file and check manually.

**Step 3: Fill in the Summary table at the bottom of evaluate.md**

Fill in Score and Critical Violations for each page. Tally the total.

**Step 4: If ANY critical violations exist**, add them to the "Violations Found" section of evaluate.md with format:
```
- [ ] [filename] [Rule ID]: [exact description of what was found vs what was expected]
```

**Step 5: If violations exist**, apply the refinement loop from docs/refinement-loop.md:
- Classify each violation (missing rule / ambiguous rule / conflicting rules / AI inference)
- For simple AI-inference violations: update the relevant .claude/rules/*.md to make the constraint more prominent (e.g., add a FORBIDDEN example that matches the exact pattern found)
- For structural violations: spawn a fresh-context subagent to regenerate only the failing sample
- Re-run check-rules.sh after any rule updates
- Update evaluate.md with the new results

**Step 6: Write the final verdict line at the bottom of evaluate.md:**
```
**Verification date:** [date]
**Final verdict:** PASS — zero critical violations / FAIL — [N] critical violations remain
**Automated check exit code:** [N]
```

Per D-13: this evaluation happens in a fresh context relative to sample generation (plan 02 ran first, this is a separate execution context).
