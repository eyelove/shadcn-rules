After this phase, the rule system is verified and the improvement process is formalized.

<decisions>
## Implementation Decisions

### Automated Violation Detection (VERF-01)
- **D-01:** Extend existing `scripts/check-rules.sh` (currently 18 checks) to cover all Phase 2+3 rules: form structure patterns, naming conventions, page template structure.
- **D-02:** Target: ~25-30 checks total, runs in <30 seconds.
- **D-03:** check-rules.sh must exit with violation count as exit code for CI integration.

### Sample Page Generation (VERF-02)
- **D-04:** Regenerate all 4 sample pages (list/dashboard/detail/form) in fresh AI context using ONLY the complete rule set (all 8 rule files).
- **D-05:** Delete old tests/samples/ and regenerate from scratch — old samples predate Phase 2+3 rules.
- **D-06:** Each sample must be generated by a subagent with fresh context (VERF-05: no self-evaluation).

### Evaluation Checklist (VERF-03)
- **D-07:** Create `scripts/evaluate.md` — a structured checklist for human review of generated samples.
- **D-08:** Format: rule → expected output → actual output → pass/fail per sample page.
- **D-09:** Cover all major rule categories: component hierarchy, tokens, forbidden patterns, forms, naming, page templates.

### Refinement Loop (VERF-04)
- **D-10:** Create `docs/refinement-loop.md` — documents the process for ongoing rule improvement.
- **D-11:** Process: generate sample → run check-rules.sh → review with evaluate.md → identify violations → update rules → regenerate → repeat.
- **D-12:** Include "when to add a new rule" criteria and "when to remove a rule" criteria.

### Fresh Context Evaluation (VERF-05)
- **D-13:** Sample generation and evaluation must happen in separate agent contexts — never evaluate in the same context that generated the code.

### Claude's Discretion
- Specific new checks to add to check-rules.sh beyond the minimum
- Evaluation checklist detail level
- Whether to add a "score" system to the evaluation (e.g., 0-100)

</decisions>
