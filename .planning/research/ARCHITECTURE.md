## Standard Architecture

This is a rule document system, not a runtime application. The "architecture" describes how files relate to each other, how AI tools consume them, and how the feedback loop operates.

### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| `CLAUDE.md` | Universal session-level rules — what applies everywhere | Root Markdown, read once per Claude Code session |
| `.claude/rules/*.md` | Domain-scoped rules — loaded conditionally by file path | Frontmatter with `paths:` targeting, e.g. `src/**/*.tsx` |
| `tokens.css` / `globals.css` | Design token definitions — single source of truth for colors, spacing, radii, typography | CSS custom properties under `:root`, mapped to Tailwind via `@theme` |
| Composed component layer | Wrapper components AI is allowed to use; encapsulates shadcn primitives + design decisions | `components/blocks/` or `components/composed/` — the only layer AI touches |
| Primitive component layer | Raw shadcn/ui components — AI should not use directly | `components/ui/` — owned by shadcn CLI, off-limits to AI generation |
| Sample pages | Reference implementations that prove rules produce correct output | `samples/` — generated by AI following rules, evaluated for conformance |
| Validation scripts | Automated checks catching rule violations in generated code | ESLint custom rules, grep scripts, or shell checks run post-generation |

## Recommended Project Structure

```
shadcn-rules/
├── CLAUDE.md                    # Universal rules (root) — short, always loaded
├── .cursorrules                 # Cursor-compatible variant (can be generated from CLAUDE.md)
├── AGENTS.md                    # Tool-agnostic variant for OpenAI Codex etc.
│
├── .claude/
│   └── rules/
│       ├── components.md        # Component usage rules (paths: src/**/*.tsx)
│       ├── tokens.md            # Design token usage rules (paths: src/**/*.{tsx,css})
│       ├── forms.md             # Form hierarchy rules (paths: src/**/form*.tsx)
│       ├── layout.md            # Page skeleton / layout rules
│       └── forbidden.md        # Explicit anti-patterns (no inline styles, no hardcoded colors)
│
├── rules/                       # Human-readable rule source (canonical, version-controlled)
│   ├── 00-overview.md           # System overview and how to use these rules
│   ├── 01-component-hierarchy.md # 3-tier model: Primitive → Composed → Page
│   ├── 02-design-tokens.md      # Token naming, usage, forbidden raw values
│   ├── 03-page-skeletons.md     # Page type templates (list, detail, settings, dashboard)
│   ├── 04-forms.md              # FormFieldSet → FormField → Input hierarchy
│   └── 05-forbidden-patterns.md # Anti-patterns with examples
│
├── tokens/
│   ├── globals.css              # :root CSS variables (color, spacing, radius, typography)
│   └── tailwind.config.ts       # Tailwind @theme mappings from CSS variables
│
├── samples/                     # AI-generated pages for validation (proof of rule effectiveness)
│   ├── list-page/
│   │   ├── page.tsx
│   │   └── README.md            # What rules were in effect, what was evaluated
│   ├── detail-page/
│   ├── settings-page/
│   └── dashboard-overview/
│
└── validation/
    ├── check.sh                 # Shell script: grep for forbidden patterns
    ├── .eslintrc.design.js      # Custom ESLint rules for design system enforcement
    └── checklist.md             # Manual visual inspection checklist
```

### Structure Rationale

- **`CLAUDE.md` (root, short):** Claude Code reads this every session. Keep under 200 lines. Universal rules only — "what applies to all files in all contexts."
- **`.claude/rules/`:** Modular rules loaded only when path patterns match. Prevents priority saturation where unrelated rules consume context. Each file owns one concern.
- **`rules/` (human-readable source):** The canonical, authored source for all rules. AI rule files are derived from or reference these. Version-controlled, reviewed as code.
- **`tokens/`:** Single source of truth for design tokens. All color, spacing, radius, typography lives here. Components reference tokens only — never hardcoded values.
- **`samples/`:** Each sample directory documents which rules governed generation and what was evaluated. This creates a reproducible test case.
- **`validation/`:** Separate from rules — the automated enforcement layer. Check scripts run after AI generates code to catch violations before they propagate.

## Architectural Patterns

### Pattern 1: Modular Rules with Path Targeting

**What:** Split rule content across multiple `.md` files in `.claude/rules/`, each with YAML frontmatter specifying which file paths trigger that rule set.

**When to use:** This project — any time rule content exceeds ~200 lines in a single file or when different rule domains apply to different file types.

**Trade-offs:**
- Pro: Claude only loads relevant rules for the current file — better signal-to-noise
- Pro: Each rule file is independently maintainable
- Con: Requires discipline to keep `CLAUDE.md` thin; teams sometimes duplicate universal rules in per-domain files

**Example:**
```yaml
---
paths: src/**/*.tsx
---
# Component Usage Rules
- Only import from components/blocks/ and components/composed/
- Never import directly from components/ui/
```

### Pattern 2: 3-Tier Component Isolation

**What:** Enforce a strict layer boundary — Primitive (shadcn `ui/`) is off-limits for AI, Composed (`blocks/`) is the AI's only entry point, Page assembles Composed components only.

**When to use:** This is the core pattern for this project. The goal is that AI sees only the composed layer's API, not the primitive layer's options.

**Trade-offs:**
- Pro: Eliminates per-page design decisions — AI has no choice but to use established components
- Pro: Composed layer captures intentional design decisions once, not per-generation
- Con: Requires maintaining the Composed layer; it must stay current with actual design needs
- Con: If Composed layer is incomplete, AI will find workarounds (document edge cases explicitly)

**Example:**
```
# Forbidden
import { Button } from "@/components/ui/button"  // raw primitive

# Required
import { ActionButton } from "@/components/blocks/ActionButton"  // composed
```

### Pattern 3: Executable Rule Enforcement (Dual Layer)

**What:** Rules exist at two levels — instructional (CLAUDE.md/rules, read by AI before generation) and executable (ESLint/grep, run after generation). Both layers must agree.

**When to use:** Any rule that is critical enough to require automated verification, not just AI-stated intent.

**Trade-offs:**
- Pro: Automated enforcement catches AI drift — when the AI forgets a rule or applies it inconsistently
- Pro: Executable rules are self-documenting: the grep pattern IS the rule definition
- Con: Requires maintaining two representations of each rule; they can drift from each other

**Example (grep check):**
```bash
# check.sh — detect inline style usage
if grep -r "style={{" src/; then
  echo "VIOLATION: inline styles found"
  exit 1
fi
```

### Pattern 4: Sample-Evaluate-Refine Loop

**What:** Rules are tested by asking AI to generate a sample page from the rules alone, then evaluating the output visually and programmatically, then updating rules based on observed drift.

**When to use:** Whenever a rule is added or changed, and at regular intervals to catch accumulated drift.

**Trade-offs:**
- Pro: Validates that rules actually produce the intended output — not just that they are written correctly
- Pro: Surfaces ambiguous rules (AI interprets them differently than intended)
- Con: Requires human evaluation effort — automated checks catch structural violations but not visual ones

## Data Flow

### Rule Authoring → AI Consumption Flow

```
rules/*.md (human-authored)
    │
    ├──► CLAUDE.md (universal subset, manually maintained)
    │
    ├──► .claude/rules/*.md (path-scoped subsets, manually maintained)
    │
    └──► .cursorrules / AGENTS.md (generated or manually maintained for other tools)

                  ↓
        [AI Tool reads at session start]
                  ↓
        AI generates code for sample page
                  ↓
        validation/check.sh + ESLint run
                  ↓
        violations → back to rules/*.md for refinement
```

### Feedback Loop Flow

```
[Author Rule]
    ↓
[AI generates sample page using rules]
    ↓
[Automated check: grep/ESLint violations?]
    │
    ├── YES: Rule violated → Is rule unclear or too weak?
    │           ↓
    │         [Strengthen rule language or add to forbidden.md]
    │
    └── NO: Run visual inspection
                │
                ├── Looks wrong → Is there a missing rule?
                │       ↓
                │     [Add new rule covering gap]
                │
                └── Looks right → Rule is validated
                        ↓
                      [Document in sample README.md]
```

### Design Token Flow

```
tokens/globals.css  (:root CSS variables)
    │
    ▼
tokens/tailwind.config.ts  (maps CSS variables to Tailwind utility names via @theme)
    │
    ▼
rules/02-design-tokens.md  (rule: "use only token names, never raw values")
    │
    ▼
.claude/rules/tokens.md  (AI instruction: "only var(--token-name) or token-mapped classes")
    │
    ▼
AI uses `text-primary` not `text-[#1a2b3c]`
    │
    ▼
validation/check.sh  (grep for hardcoded hex/rgb values as violation signal)
```

## Scaling Considerations

This is a personal-use rule document system. "Scaling" means complexity growth, not user growth.

| Scale | Architecture Adjustments |
|-------|--------------------------|
| 1 dashboard type | Single CLAUDE.md is sufficient; rules/ directory is premature |
| 3-5 dashboard types | Introduce `.claude/rules/` modular split by domain (components, tokens, forms) |
| 10+ page patterns | Add `samples/` directory with one reference per page type; validation scripts become essential |
| Multi-team or public | Extract rules to separate package; generate tool-specific files (CLAUDE.md, .cursorrules) from a canonical source |

### Scaling Priorities

1. **First bottleneck:** CLAUDE.md grows too long → Claude prioritizes poorly. Fix: split into `.claude/rules/` with path targeting.
2. **Second bottleneck:** Rules reference Composed components that don't exist yet → AI improvises. Fix: build Composed library alongside rules.

## Anti-Patterns

### Anti-Pattern 1: Monolithic Rule File

**What people do:** Put all rules — component hierarchy, token usage, form patterns, forbidden patterns — into one long CLAUDE.md.

**Why it's wrong:** Claude Code loads the entire file at high priority regardless of context. Long files cause priority saturation: everything competes for attention equally, so critical rules get diluted. Files over ~300-400 lines show measurably worse adherence.

**Do this instead:** Keep root CLAUDE.md under 200 lines with universal rules. Domain-specific rules go in `.claude/rules/` with `paths:` frontmatter targeting.

### Anti-Pattern 2: Rules Without Validation

**What people do:** Write comprehensive rules, verify by reading them (not by running AI against them), then use in production without feedback.

**Why it's wrong:** Rules that read well often produce unexpected AI output. AI interprets ambiguous phrasing differently than the author. "Use composed components" means nothing if AI doesn't know which components exist.

**Do this instead:** After writing or changing any rule, generate a sample page from the rules alone. Run automated checks. Compare output against intent. Update rules based on what the AI actually did.

### Anti-Pattern 3: Rules Reference Composed Components That Don't Exist

**What people do:** Write rules that say "use `<DataTable>` component" before building `<DataTable>`, expecting AI to just wait.

**Why it's wrong:** AI will not wait — it will create its own implementation using raw HTML or shadcn primitives, exactly the problem the rules were meant to prevent.

**Do this instead:** Build the Composed component layer and the rules that reference it in lockstep. Rules can reference a component only after a stub (or full implementation) exists in the sample project.

### Anti-Pattern 4: Separate Rules for Each AI Tool Maintained Independently

**What people do:** Write CLAUDE.md, .cursorrules, and AGENTS.md as independent files with slightly different content.

**Why it's wrong:** They drift. Cursor says "use Vitest", CLAUDE.md says "use Jest." AI tools produce inconsistent outputs depending on which tool was used.

**Do this instead:** Maintain one canonical source in `rules/` directory. Generate or manually sync CLAUDE.md, .cursorrules, and AGENTS.md from it. Flag all three files in git as "derived from rules/."

## Integration Points

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| `rules/` → `CLAUDE.md` | Manual sync, human-curated subset | rules/ is the source; CLAUDE.md is derived; keep CLAUDE.md thin |
| `rules/` → `.claude/rules/*.md` | Manual decomposition with path targeting | Same source, broken into path-scoped files |
| `tokens/globals.css` → `rules/02-design-tokens.md` | Rule references CSS variable names from token file | Token file must be committed before rule can reference it |
| `validation/check.sh` → `rules/05-forbidden-patterns.md` | Grep patterns mirror forbidden rule language | These two must be kept in sync; if one changes, so must the other |
| `samples/` → rule refinement | Visual + automated evaluation drives rule updates | Samples are test artifacts, not production code |

### Build Order (Component Dependencies)

This is a documentation/rule project, so "build order" means authoring order — what must exist before what can reference it.

```
1. tokens/globals.css             ← Define all CSS variables first
   tokens/tailwind.config.ts      ← Map to Tailwind utility names

2. rules/01-component-hierarchy.md ← Define 3-tier model
   rules/05-forbidden-patterns.md  ← Define anti-patterns

3. Composed component stubs        ← Minimal implementations so rules can reference them
   (components/blocks/ in sample project)

4. rules/02-design-tokens.md       ← Now has actual token names to reference
   rules/03-page-skeletons.md      ← Now has actual composed components to reference
   rules/04-forms.md

5. CLAUDE.md (root)                ← Synthesize universal rules from rules/
   .claude/rules/*.md              ← Decompose into path-targeted files

6. samples/ generation             ← AI generates pages using completed rules

7. validation/check.sh             ← Codify observed violations as grep checks
   validation/.eslintrc.design.js  ← Codify structural checks as ESLint rules

8. Feedback loop iteration         ← Rules → sample → validate → refine → repeat
```

## How AI Tools Consume Rule Files

**Claude Code (HIGH confidence — verified with official docs):**
- Reads `CLAUDE.md` at session start (recursive: root + parent dirs + `~/.claude/CLAUDE.md`)
- Reads `.claude/rules/**/*.md` automatically, loaded with same priority as CLAUDE.md
- Rules with `paths:` YAML frontmatter only activate when working in matching files
- 6 levels of hierarchy: user-global → project-root → subdirectory → rules/ → path-targeted rules → imports

**Cursor (MEDIUM confidence — docs reviewed, no direct test):**
- Reads `.cursorrules` (single file, project root only)
- No hierarchy, no path targeting, no auto-memory
- Context window limit applies — files >800 lines show degraded adherence
- Newer versions also read AGENTS.md

**AGENTS.md (MEDIUM confidence — multiple sources agree, format is relatively new):**
- Tool-agnostic format, read by Claude Code, OpenAI Codex CLI, and others
- Supports subdirectory placement for monorepo-style scoping
- Markdown format, no special frontmatter support (unlike `.claude/rules/`)

**Recommendation for this project:** Author rules in `rules/` directory, maintain `CLAUDE.md` + `.claude/rules/` as primary target (Claude Code is the primary tool described in PROJECT.md), and maintain `.cursorrules` as a flattened fallback for Cursor compatibility.
